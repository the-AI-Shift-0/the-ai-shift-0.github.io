Master Trove Production Standard

The complete standard for Demo 2 — from form design to email delivery. Derived from Sales Analyzer + AI Caller as the canonical reference implementations. Every future video follows this exactly.

Established: 2026-05-26  |  Based on Sales Analyzer + AI Caller (Videos 1 & 2)

Section 1: What is the Master Trove?
The Master Trove is Demo 2
Every video in the channel has two components. Demo 1 is the sophisticated simulation only Sai can build with Claude Code. Demo 2 is the viewer deliverable — a personalized Master Trove they receive via email after filling out a form. This is what they walk away with. It is NOT generic. It is personalized to their specific business profile.
Value Why a Master Trove is worth an email address
The Master Trove is personalized to each viewer's business. It references their industry, revenue range, team size, and the specific operational challenge they shared in the form. That personalization is what makes it valuable enough to give up an email address for. A generic cheat sheet is noise. A document that says "Here is the exact setup for a $3M SaaS company doing outbound calling with 2 reps" is useful immediately.
Structure Master Trove pipeline: form → API → email
The viewer fills a two-step form on the website. Step 1 captures name, email, and phone (optional). Step 2 asks business-specific qualifying questions. On submit, the form POSTs to Railway API, which calls Claude to generate 3-section personalized content, subscribes the user to a Kit email sequence, and sends the output via email. The entire pipeline completes in 15-30 seconds. No manual work. No handoff between systems.
Section 2: Pipeline Architecture
Why we made these infrastructure decisions. Do not re-debate them — they are settled (May 2026).
Public form (theaishift.biz/tools/[video]/master-trove/)
↓ POST /generate/[video-slug]
Railway API (master-trove-api-production.up.railway.app)
↓ Anthropic Claude (claude-sonnet-4-6)
↓ Kit API (subscriber + custom fields + sequence subscription)
Personalized email → subscriber's inbox
Why Railway Vercel times out on Claude API calls
Vercel functions have a 10-second timeout on free tier, 60 seconds on Pro. Claude API calls (especially with long context) take 15-30 seconds. Railway handles this natively with no timeout constraints. Railway also costs $5/month fixed. Settled May 2026.
Why Kit Kit has a full API. Mailchimp does not.
Kit's API allows us to: (1) subscribe users to sequences programmatically, (2) set custom fields on the subscriber profile, (3) add tags for segmentation, (4) trigger automations. All of this is scriptable from our server. Mailchimp has limited API coverage. We never need to open Kit's dashboard for a new video launch — everything is code.
Why Direct Sequence Subscription The server subscribes users. No Kit UI automations ever.
The API flow is: form → generate → subscribe → email sent. The server calls Kit API directly with the sequence ID, email, and custom fields. Kit handles the rest via pre-configured email sequences. We never create automations in Kit's dashboard. Automations are implicit — the sequence exists, the API adds the subscriber, Kit sends the email. This keeps video launches fast: create form → add prompt builder → add sequence ID → done.
Section 3: Form UX Standards
These apply to every Master Trove form. Non-negotiable. Reference implementations: Sales Analyzer and AI Caller forms.
Structure Two-step form, progress indicator, one button click per step
Step 1 (Contact): Name, email, mobile number (optional). Email is required first — abandoned users get re-targeted. Label phone field "Mobile number (optional)".

Step 2 (Qualify): Business-specific questions — industry, revenue, and video-specific fields like call volume or sales cycle.

Progress indicator: Show "Step 1 of 2" and "Step 2 of 2" visibly at top of form. On "Next" button, no page reload — just Step 1 → Step 2 transition.
Required Industry field - always present with custom input fallback
Dropdown options: Online coaching/education, E-commerce, Professional services, Real estate, Healthcare, SaaS, Other.

When "Other" is selected: A text input appears immediately below. Placeholder: "Tell us your industry (e.g. AC unit repair)". The custom text replaces "Other" in formData before submission.

Validation: Required. No submission without an industry selected and (if Other) custom text filled in.
Required Multi-select fields: use checkboxes, not radios. Add open-text fallback.
Any field where multiple answers are valid — triggers, goals, recording tools, pain points — use checkboxes. Single-answer fields (deal size, revenue range, call volume) stay as dropdowns.

Open-text fallback: Below every multi-select group, add a textarea. Placeholder: "Anything else? Add notes here..." Users can add things we didn't predict. This prevents losing data because we didn't think of an option.

Validation: Multi-select groups require at least one checkbox checked.
Validation Fail visibly. Never proceed with missing required fields.
Required fields: industry (or custom text if Other), revenue, and any video-specific required fields. On validation failure, show an alert describing what's missing. Do NOT proceed to processing state until all required fields are filled and valid. On form submission, validate before sending to Railway.
Processing State Hide form. Show spinner. Wait for Railway.
On successful validation and submit click: (1) Hide the form immediately. (2) Show a spinner with message (e.g. "Building your Master Trove..."). (3) POST to Railway. (4) Wait for response. Do NOT show a success screen until Railway responds successfully. If Railway returns an error, show an error state. Never show success if the API call failed.
Error State Network or API errors are handled gracefully
If Railway call fails (network timeout, HTTP error, any exception): Hide spinner. Show error message: "Something went wrong. Please try again or email hello@theaishift.biz." Show a "Try Again" button that restores the form to Step 2 (the last filled step). NEVER show the success screen if Railway returned an error.
Success State Outcome language only. No jargon. No technical details.
Structure:
✓ icon → headline ("Your [Video Name] Master Trove is being built") → 1 sentence about personalization and timing ("It usually lands in your inbox within 60 seconds") → inbox tip (check spam/promotions folder) → "What's on its way" bullet list (3 bullets, outcome-focused) → fallback line ("Didn't receive it? Email us at hello@theaishift.biz and we'll sort it.")

Language: Use outcome language only. No "system prompt", no "prompt", no "Claude", no "ChatGPT". No technical implementation details. Example (AI Caller): "A diagnosis of what's costing your team the most calls right now" not "Your system prompt". The bullets describe VALUE, not technical contents.
Section 4: Output Format Standards
What Claude generates. This is the structure every prompt must enforce.
Structure 3 sections. Always. STEP 1, STEP 2, STEP 3.
Every Master Trove has exactly 3 sections. Each section header is in ALL CAPS: STEP 1: [Section Name], STEP 2: [Section Name], STEP 3: [Section Name]. The section names are video-specific (set by the prompt builder), but the structure never varies.
Context Line Each section opens with one sentence explaining what it is
After the section header, one blank line, then a context line. One sentence. Plain English. Tells the viewer what this section IS and why it matters. Example (AI Caller): "Here is the AI calling platform best matched to your setup, along with what you will spend monthly." Then a blank line, then the section content.
Format Plain text only. No markdown. No asterisks, no #, no ``
The Claude prompt must explicitly enforce this: "Write in plain text only. No markdown, no asterisks, no hashtags." No # headers, no ** bold, no bullet markdown. Use plain dashes or numbered lists instead. Section headers (STEP 1 / STEP 2 / STEP 3) are the only formatting allowed. All output is readable as plain text in an email client.
Personalization Every section references the viewer's specific answers
Do not write generic advice. Every section must reference the viewer's form data. The section 1 recommendation (platform, framework, approach) must be justified specifically for their industry + their stated challenge. Example: "For a $2M e-commerce business doing post-purchase follow-up calls, Retell AI is the best fit because..." not "Retell AI is the best platform..."
Length Max 2000 tokens. Enough for 3 meaty sections.
The Claude API call sets max_tokens: 2000. This is plenty for 3 substantial sections without overwhelming the reader. Each section should be ~500-700 tokens.
Section 5: CRM / Kit Field Standards
Every form submission populates Kit subscriber fields. This is the data structure and convention.
Standard Fields Every video stores these fields on the Kit subscriber profile
master_trove_content — The Claude-generated output (truncated to 3000 chars, plain text)
master_trove_video — The video slug (e.g. 'sales-analyzer')
industry — The viewer's industry
revenue — Annual revenue range
phone — Mobile number (optional, empty string if not provided)
Video-Specific Fields Mapped in server.js FIELD_MAP per video
Sales Analyzer: sales_reps, deal_size, sales_cycle, call_recorder, coaching_problem (biggest pain), top_objection, platform

AI Caller: call_volume, call_trigger, call_goal

Future videos: Add new fields to FIELD_MAP in server.js. Create the Kit custom fields BEFORE the first form submission, or they are silently ignored.
Convention Form camelCase → Kit snake_case. Arrays as comma-separated strings.
Naming: Form field name (camelCase) maps to Kit field key (snake_case). Example: repCount → sales_reps.

Arrays: Multi-select fields (checkboxes) are stored as comma-separated strings in Kit. Example: ["outbound", "inbound"] becomes "outbound,inbound".

FIELD_MAP: Define once in server.js, reuse for every submission. The server reads FIELD_MAP and copies formData fields to Kit using these mappings.
Sequence Subscription KIT_SEQUENCE_IDS maps video slug to Kit sequence ID
KIT_SEQUENCE_IDS = { 'sales-analyzer': 'seq_123', 'ai-caller': 'seq_456', ... }

On form submission, the server looks up the sequence ID for the video, then calls Kit API to subscribe the user. The user is added to the sequence immediately. Kit's pre-configured email automations handle the rest. The server never opens Kit's dashboard.
Section 6: Adding a New Video - Checklist
Exact steps in order. Follow this every time a new video launches.
  1. Create Kit custom fields — Via API or node scripts/kit-setup.js. Create all video-specific fields listed in the FIELD_MAP before any form submission.
  2. Write prompt builder — railway-server/prompts/[slug].js. Returns {systemPrompt, userMessage}. Enforces 3 sections, context lines, plain text, personalization. Max 2000 tokens.
  3. Write email template — railway-server/emails/[slug]-email.js. Returns HTML email with the generated content rendered.
  4. Add sequence ID to KIT_SEQUENCE_IDS — In server.js, add '[slug]': '[kit-sequence-id]'.
  5. Register prompt builder — In server.js promptBuilders map, add '[slug]': require('./prompts/[slug].js').
  6. Register email template — In server.js emailTemplates map, add '[slug]': require('./emails/[slug]-email.js').
  7. Add FIELD_MAP — In server.js, map all form field names (camelCase) to Kit field keys (snake_case).
  8. Create form HTML — public/tools/[slug]/master-trove/index.html. Follow form UX standards above. Two-step form, progress indicator, validation, spinner, success/error states.
  9. Test form submission — Submit with test data. Verify Kit subscriber has all fields populated correctly.
  10. Test email delivery — Verify personalized email arrives with content rendered correctly.
  11. Push Railway server — Auto-deploys to production. All code changes live.
  12. Push website — GitHub Pages deployment. Form is live at theaishift.biz/tools/[slug]/master-trove/.

Section 8: Mandatory Feasibility Check
Before writing the prompt builder or form, verify technical feasibility. Demos must be deliverable.
Do Verify the demo can work in the form + Claude workflow
Before building, ask: "Can Claude generate this output in 2000 tokens? Can we personalize it to their industry + pain point? Can we deliver it in an email as plain text?" If the answer to any is no, the demo scope is too large — cut it or split it.
Never Build a Master Trove that requires user interaction or multi-step process
The deliverable is plain text in an email. It cannot require the viewer to fill out a form, click a link, or take additional steps. If the output needs interactivity, it should be Demo 1 (Claude Code custom interface), not Demo 2 (Master Trove).
Never Ask for form fields you cannot use in the prompt
Every form field must be referenced in the Claude prompt for personalization. Do not ask "What color is your logo?" if the prompt never uses it. Every field = more friction on the form. Only ask what drives the output.
