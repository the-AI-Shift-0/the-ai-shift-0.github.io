The AI Shift — Channel Standard

Demo Spec Standard

Two parts: (1) standing rules that never change — every demo follows these automatically. (2) what Sai fills out per video — the only things that vary.

Last updated: 2026-04-29  |  Established after Video 1 (AI CEO) post-mortem

Part 1 — Standing Rules
These are permanent channel decisions. They apply to every video, every demo. Sai does not need to specify any of these in his script — they are already decided. Anyone building or editing demos must know these.
Structure Every video has two demo types — always
Demo 1 (Internal Build): What Sai actually built using Claude Code. Sophisticated, polished, impressive. Viewers watch it — they cannot replicate it themselves. This is the belief-breaker: shows what AI is capable of at a high level.

Demo 2 (Viewer Deliverable): What the viewer walks away with. Built on a web chat interface (Claude.ai or ChatGPT). Simplified to 70-80% of Demo 1's value with minimum friction. This is the action-driver: something they can use immediately.
Interface Demo 1 = Claude Code custom interface. Demo 2 = web chat only.
Demo 1 always shows the custom interface built via Claude Code — Sai will know exactly what this looks like for each video because he built it. Demo 2 always uses a standard web chat interface (Claude.ai or ChatGPT chat). Never show APIs, terminals, code environments, or anything requiring technical setup as the thing the audience replicates. The audience is $1-10M SMB owners using web chat tools, not builders.
Bridge Rule Demo 2 must deliver 70-80% of Demo 1's value via web chat
When designing Demo 2, the goal is maximum value for the viewer at minimum friction. If a Demo 2 requires more than pasting a prompt or filling a form, it needs redesigning. The bridge from what Claude Code can do to what a web chat user can do is AP's responsibility to figure out — Sai defines the outcome, AP designs the simplest path to it.
Deliverable Standard two-stage opt-in form + personalized file — every video
Every video uses the same delivery mechanism: a multi-step form where step 1 captures name and email (for abandonment recovery), step 2 asks qualifying questions that personalise the output, and delivery is automatic on completion. The viewer receives a personalized file via email. AP recommends this to Sai, Sai approves, AP builds it. Sai defines only what the file contains. AP designs the form questions based on that.
Data & Privacy Real data is blurred. Illustrative data is used where real data would expose PII.
Real business numbers (revenue, costs, units) are shown blurred to add authenticity without exposing sensitive figures. Names, client names, and team members are always blurred or replaced with illustrative placeholders. Production applies this — AP takes a call on what's real vs illustrative.
Recording Heavy lifting happens at recording, not in editing
The goal is to capture demos cleanly at recording time. Demos should not rely on editing to fix weaknesses. If a live demo is unpredictable (depends on AI output quality in the moment), it should be pre-recorded and polished before the shoot. Editing magic is a tool, not a crutch.
AI Tool Stay AI-agnostic. Do not push Claude or ChatGPT over each other.
Do not advocate for one AI tool over another in the video. Viewers have their preferences and creating friction or polarization hurts conversions. Only specify a tool on screen if there is a genuine functional reason (e.g. voice mode, file upload, a specific capability). The deliverable should work in whatever AI tool the viewer already uses.
Part 2 — What Sai Fills Out Per Video
Everything above is already decided. These are the only things that change video to video. Sai drops one block per demo scenario into his script. If all fields are filled, no follow-up call is needed.
Field Why It's Video-Specific
Business problemOne sentence. Confirms the demo is solving the right pain for this video's story.
Exact inputThe prompt or voice input word-for-word. Different for every use case. Used directly in simulation and dry-run testing.
Expected outputWhat the AI produces. What appears on screen. Varies by use case — builder cannot guess this.
Screen flowNumbered list of screens in order. Different for every video. Drives the demo simulation and editor capture list.
Wow momentThe single on-screen beat that must hit hardest. Unique to each video's demo. Production gives this the most polish.
Deliverable substanceWhat is actually inside the personalized file. The format and delivery are standard — the content is video-specific.
The Template
Copy one block per demo scenario into the script. A video typically has 2-3 demo scenarios (e.g. sales, ops, team) — each gets its own block. All fields are required.
